# データ辞書

## データ辞書記載内容

### 1. 基本情報（その用語の正体を明かす）

まずは、その用語が一般的・システム的にどう呼ばれるかを定義します。

* **用語名（論理名）**: 日本語での正式名称（例：顧客、注文合計金額）。
* **読み方**: 読み間違いを防ぐため（例：月極（げっきょく）、既存（きぞん）など）。
* **英名（物理名）**: データベースのカラム名やソースコードで使う名前（例：`customer_id`, `total_amount`）。
* **略称・別名**: 現場で使われている通称（例：「注文」を「オーダー」と呼ぶ場合など）。
* **定義（説明文）**: その用語の具体的な意味。**「誰が読んでも同じ解釈になる」**ように書きます。

---

### 2. ビジネスルール（境界線を引く）

ここが最も重要です。言葉の「範囲」と「条件」を厳密に定義します。

* **包含範囲（何を含み、何を含まないか）**:
  * 例：「顧客」とは、会員登録済みの人を指すのか？ 退会した人も含むのか？ 見込み客は？
* **算出根拠（計算式）**:
  * 例：「注文合計金額」は、税込か税抜か？ 送料や手数料は含むのか？ ポイント利用分は引くのか？
* **ライフサイクル（状態の変化）**:
  * 例：「注文」には、「未入金」「発送済み」「キャンセル」などの状態があることを明記します。
* **制約・バリデーション**:
  * 例：「商品コード」は必ず大文字英字2桁＋数字4桁である、など。

---

### 3. 管理・技術情報（運用に備える）

実際にシステムを構築・運用する際に必要な情報です。

* **データ型・桁数**:
  * 例：数値（10桁）、文字列（最大100文字）など。
* **更新権限（誰が管理するか）**:
  * 例：このデータは「顧客自身」が更新するのか、「営業担当者」がシステムから入力するのか。
* **機密性（セキュリティレベル）**:
  * 例：個人情報（高）、社外秘（中）、公開情報（低）など。
* **元ネタ（ソース）**:
  * 例：このデータはどの画面から入力されるか、または外部のどのシステムから連携されるか。

---

### 具体的な定義例（例：注文金額）

用語定義をまとめると、以下のような形式になります。

| 項目 | 定義内容 |
| :--- | :--- |
| **用語名** | 注文合計金額 |
| **英名** | `total_order_amount` |
| **定義** | 1回の注文において、顧客が最終的に支払うべき総額。 |
| **算出式** | Σ(各商品の単価 × 数量) + 消費税 + 送料 - クーポン割引額 |
| **範囲** | キャンセルされた注文は含まない。 |
| **データ型** | 数値 (12桁) |
| **備考** | 2023年10月のインボイス制度対応により、税計算の端数処理ルールを遵守すること。 |

---

### 用語定義を行う際のコツ

* **「同じ意味の別名」を統合する（同義語）**
  * 「ユーザー」「会員」「ログインID」が同じものを指しているなら、システム上は1つの用語に統一します。
* **「違う意味の同名」を分ける（多義語）**
  * 「単価」という言葉が、仕入れ値（原価）を指すのか、販売価格を指すのかは部署によって異なります。「販売単価」「仕入単価」と明確に分けます。
* **「否定形」で範囲を決める**
  * 「～は含まない」と書くことで、境界線が非常にクリアになります。
